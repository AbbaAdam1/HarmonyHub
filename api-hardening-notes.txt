HarmonyHub API Hardening Notes
Date: 2026-03-24
File changed: src/pages/api/deezer/album/[id].ts

Summary
These changes improve reliability, security, and production behavior for the Deezer album API route.

What changed and why it is better

1) Method guard (GET only)
- Added a request method check that rejects non-GET requests with 405 and sets Allow: GET.
- Better because it prevents accidental or invalid usage patterns and makes endpoint behavior explicit.

2) Stronger input validation
- Album id now must be a string of digits.
- Better because malformed ids are rejected early with a clear 400 error instead of sending bad upstream requests.

3) Upstream timeout protection
- Added AbortController with an 8 second timeout for Deezer fetches.
- Better because requests cannot hang indefinitely, improving user experience and protecting server resources.

4) Better error mapping
- Upstream non-OK responses now return the upstream status with a safe error message.
- Timeout returns 504.
- Unexpected server failures return 500 with a generic message.
- Better because clients receive clearer and more predictable status codes while avoiding leaking internal details.

5) Caching headers for successful responses
- Added Cache-Control: public, s-maxage=300, stale-while-revalidate=600.
- Better because repeated album requests are faster and cheaper, especially behind CDN/edge caches.

6) Timeout cleanup
- Added clearTimeout in finally.
- Better because it avoids lingering timers and reduces risk of small resource leaks.

Net effect
- More robust API route behavior under normal load and failure scenarios.
- Clearer client semantics (400/405/504/5xx) and safer production defaults.
- Better performance via cacheability for album responses.
